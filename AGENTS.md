# galay-* 仓库目录结构与构建约束模板

> 用途：适用于所有 `galay-*` 仓库的通用目录约束。新增模块时优先复用已有层级，避免目录职责重叠。

```text
<repo-root>/
├── benchmark/                 # 基准测试与性能对比用例
├── cmake/                     # CMake 模块与构建配置片段
├── docs/                      # 文档（设计、使用、API、发布说明）
├── examples/                  # 示例程序
│   ├── common/                # 示例公共代码
│   ├── import/                # 示例依赖导入/适配
│   └── include/               # 示例头文件
├── <core-module>/             # 核心库源码目录（按仓库实际命名）
│   ├── async/                 # 异步能力
│   ├── common/                # 通用组件
│   ├── concurrency/           # 并发相关实现
│   ├── kernel/                # 调度/IO 等核心内核能力
│   └── module/                # 功能模块集合
├── scripts/                   # 构建、测试、基准相关脚本
└── test/                      # 测试用例（单测/集成/场景测试）
```

## 使用约定

- `<core-module>/` 只放可复用库实现，不放业务示例和一次性实验代码。
- `examples/` 与 `test/` 分离：示例偏“教学演示”，测试偏“自动校验”。
- 设计代码架构时，先完成模块边界划分，再按模块创建目录进行物理隔离，避免多个职责混放在同一目录。
- 各目录职责与放置内容一一对应：`cmake/` 放 CMake 模块、Find 模块、可复用配置片段等；`benchmark/` 放基准可执行体与压测/对比脚本；`scripts/` 放构建/测试/基准的辅助脚本；`docs/` 放设计与说明文档；`test/` 放单元测试与可自动化回归的用例。勿把构建脚本、测试或基准逻辑散落在与职责不符的目录中。
- 所有构建临时文件、编译中间产物、生成二进制/库文件、CMake/Ninja 缓存统一放在 `build/` 目录。
- 禁止在 `<core-module>/`、`examples/`、`test/`、`docs/` 等源码目录落地任何构建产物或临时编译文件。
- `build/`、`.cache/` 属于本地产物目录，不应作为源码结构依赖，建议加入忽略规则。
- 新增一级目录前，先确认是否能归入现有 `benchmark/docs/examples/scripts/test` 等职责域。

## 对外接口与注释

- 凡对外导出的头文件、类、函数、稳定 ABI/API 面，**必须**配有清晰、可维护的文档注释：用途、前提条件、错误/异常语义、线程与生命周期约定、不保证的行为（如有）。
- 注释应随实现同步更新，避免头文件与行为不一致。

## 测试、基准与合入前验证

- 任何新增模块或逻辑变更，**应**有对应**单元测试**（可放在 `test/` 或与仓库约定一致的测试目录中）；能测的行为尽量覆盖，避免无测上线。
- 使用 CMake 编译并启用单测时，**必须**指定 `BUILD_TEST=ON`（或仓库文档中写明的等价开关），在本地可完整构建测试目标。
- 源码有实质性更新合入前，**应全量**跑通现有测试，并跑完与本次变更相关的 **benchmark**；若 benchmark 是性能门槛的一部分，全量或约定子集以仓库 `README`/`docs` 为准，但须明确记录且不得跳过对变更模块的压测面。
- 性能类 benchmark 的产出**必须**附带可复现的说明与数据，至少包括：
  - **测试机配置**：CPU 型号、核数/线程、主频、内存、OS 与内核版本、关键编译器与优化选项（如适用）。
  - **测试配置**：并发度、连接数、payload、迭代次数/时长、压测端与被测端拓扑等。
  - **结果指标**：如 QPS、RPS、延迟分位数、吞吐、错误率、CPU/内存占比等，与业务含义二选一或并列说明，避免只有裸数字无上下文。
- 以上 benchmark 说明可与代码同目录的 `README`、或 `docs/` 下专项文档、`benchmark/` 内旁注二选一，但**须能根据文档复现实验**。

## 构建系统与版本对齐

- 每次发版时，`CMakeLists.txt`/`cmake/` 中声明的项目版本必须与发版版本（Tag/Release Note/CHANGELOG）保持一致，不允许出现版本漂移。
- 使用 Bazel 的仓库必须同步维护对应版本与构建目标：`BUILD.bazel`、`MODULE.bazel`、`WORKSPACE`（或等价文件）需与 CMake 配置语义对齐。
- 新增模块、目标或依赖时，必须同时更新 CMake 与 Bazel 两套构建描述，确保两套构建系统都可独立完成编译与测试。

## 发版前检查清单

- [ ] **版本一致性**：`CMakeLists.txt`/`cmake/` 版本号、Git Tag、`CHANGELOG.md`、`docs/release_note.md` 中版本完全一致。
- [ ] **CMake 构建验证**：使用发布配置完成全量构建，并确保单测配置包含 `BUILD_TEST=ON` 且测试目标可构建。
- [ ] **Bazel 构建验证**：对应 Bazel 目标与依赖定义已同步更新，关键目标可独立构建与测试。
- [ ] **测试全量通过**：当前分支在本次发版范围内的测试全部通过（含新增/受影响模块）。
- [ ] **Benchmark 完整产出**：已运行相关压测并产出机器配置、测试配置、结果指标（QPS/RPS/延迟/吞吐等）及复现说明。
- [ ] **变更可追溯**：架构/性能关键决策均在文档中记录“背景-方案-权衡-结论”，并与代码变更一一对应。

## 角色能力约束

- 参与 `galay-*` 仓库开发与评审的工程角色，默认要求具备**精通 C++**（现代 C++20/23、模板与元编程、RAII、异常安全、对象生命周期、ABI 兼容）能力。
- 默认要求具备**高性能编程**能力：能够识别并优化热点路径，关注 cache 友好性、锁竞争、内存分配、系统调用开销、零拷贝与批处理策略，并以可复现 benchmark 数据验证收益。
- 默认要求具备**底层架构设计**能力：能进行并发模型、调度模型、I/O 模型（如 epoll/io_uring）、内存模型与模块边界设计，兼顾性能、可维护性与可扩展性。
- 所有性能或底层架构相关决策，应提供清晰设计依据、权衡说明与验证结果，避免“仅凭经验”的不可追溯改动。

## 命名风格约束

- **核心库文件命名**：`<core-module>/` 下头文件采用 `PascalCase`，如 `Runtime.h`、`IOUringScheduler.h`、`AsyncMutex.h`；新增核心模块文件保持同风格。
- **测试/示例/压测文件命名**：统一使用“编号前缀 + 语义名”模式，推荐 `T<number>-snake_case.cc`、`E<number>-snake_case.cc`、`B<number>-snake_case.cc`，如 `T10-compute_scheduler.cc`、`E1-sendfile_example.cc`、`B3-tcp_client.cc`。
- **类型命名**：类、结构体、枚举、类型别名使用 `PascalCase`，如 `TaskState`、`AwaitableBuilder`、`SchedulerCoreStage`。
- **函数命名**：函数/方法使用 `lowerCamelCase`，如 `scheduleTaskDeferred()`、`takeTaskResult()`、`setTaskScheduler()`；布尔语义优先 `is/has/can/should` 前缀。
- **成员变量命名**：类成员统一 `m_` 前缀 + `snake_case`，如 `m_state`、`m_runtime`、`m_result_storage`；原子与锁相关成员同样遵循该规则。
- **普通变量命名**：局部变量、函数参数、命名空间级内部变量采用 `snake_case`，如 `retain_ref`、`then_state`、`runtime_handle`；避免单字符命名（循环计数器等短生命周期场景除外）。
- **常量命名**：编译期常量（`constexpr`/模板常量）使用 `k` 前缀 + `PascalCase`，如 `kInlineResultBytes`；枚举值保持现有风格并在同一枚举内一致。
