## 性能总览

### 网络 IO (TCP)

> TCP benchmark 对比必须固定以下元数据再看数字：
> `backend`、`build`、`role`、`io_mode`、`scenario`。
> 推荐口径：
> - plain 路径：`B2-TcpServer` + `B3-TcpClient`
> - iov 路径：`B11-TcpIovServer` + `B12-TcpIovClient`
>   当前实现是用户自管双段 iovec（`64B + rest`），不再混入 `RingBuffer` 额外拷贝
> - 只在相同 backend、相同 build、相同连接数/消息大小/时长下比较 plain 与 iov
> - Rust/Go 对比也必须先对齐这些条件，否则结果不可直接解释

测试日期：2026-03-07

统计口径：
- 本地 `macOS / kqueue`：`3` 轮压测，主表展示中位数，范围展示 `min ~ max`
- 远端 `Linux / epoll + io_uring`：`3` 轮压测，主表展示中位数，范围展示 `min ~ max`

场景 A：`256B`，`100` 并发连接，`10s`

结论：
- 所有 TCP 结论统一基于 `3` 轮压测中位数，不再采用单轮结果下判断。
- `Mac / kqueue` 上，`iov-2seg` 已基本贴近 `plain`：`256B` 落后约 `4.5%`，`4KB` 落后约 `1.3%`。
- `Linux / epoll` 上，当前 `iov-2seg` 相比 `plain` 仍有明显差距：`256B` 约落后 `13.0%`，`4KB` 约落后 `15.4%`。
- `Linux / io_uring` 上，`256B` 里 `iov-2seg` 仍落后 `plain`，但 `4KB` 场景中位数已比 `plain` 高约 `12.4%`；同时其波动范围仍偏大，后续对 Rust / Go 时必须继续用多轮结果。

> **注意**: 不同平台的文件 IO 实现方式不同，性能数据不能直接对比：
> - **kqueue / io_uring**: 使用普通文件模式，可利用操作系统页缓存，热数据访问极快
> - **epoll + libaio**: 必须使用 `O_DIRECT` 模式，绕过页缓存，每次都是真实磁盘 IO
>
> epoll + libaio 的测试反映的是**真实磁盘性能**，适合大文件顺序 IO 场景（数据库、日志系统）。
> 测试参数：`-w 1 -batch 16`（单 worker，批量 16）

详细数据: [13-文件IO.md](13-文件IO.md#性能压测数据)

## 关键结论

### 网络 IO
- `Mac / kqueue`: `256B plain` 中位数 `242,874 QPS`，`4KB plain` 中位数 `219,922 QPS`
- `Linux / epoll`: `256B plain` 中位数 `114,173 QPS`，`4KB plain` 中位数 `101,868 QPS`
- `Linux / io_uring`: `256B plain` 中位数 `135,894 QPS`，`4KB iov-2seg` 中位数 `112,403 QPS`
- 当前 TCP 结果均按统一测试环境口径整理，且统一使用 `3` 轮中位数；不再引用单轮峰值
- 横向比较前先确认 `platform/backend/build/io_mode/role/scenario` 完全一致

### 文件 IO
- 🚀 **kqueue (macOS)**: 38,095 IOPS
- 🚀 **io_uring (Linux)**: 20,000 IOPS
- 🚀 **epoll (Linux)**: 20,000 IOPS

### ComputeScheduler
- ⚡ **macOS (M4)**: 400万+ tasks/s，延迟 7.01 μs
- ⚡ **Linux 历史数据**: 135万+ tasks/s（待补环境信息）

### MpscChannel
- ⚡ **macOS (M4)**: 2777万+ msg/s
- ⚡ **Linux 历史数据**: 523万+ msg/s，批量接收 1470万+ msg/s（待补环境信息）
- 适合跨线程 / 跨调度器单消费者场景

### UnsafeChannel
- ⚡ **macOS (M4)**: 1098万+ msg/s，延迟 5.96 μs
- 适合同线程 / 同调度器极低开销通道场景

### SendFile 零拷贝传输
- 🚀 **epoll (Linux)**: 181.49 MB/s，比 read+send 快 78%
- 🚀 **io_uring (Linux)**: 141.24 MB/s，比 read+send 快 169%

### 生产就绪度
- ✅ 所有平台零错误率
- ✅ 稳定性优秀
- ✅ 推荐生产使用
